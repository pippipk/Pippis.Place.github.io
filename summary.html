Relevant Implications

- We have future proofed the design of this program. Future Proofing means the program is designed so new features can be added easily and the old ones easily removed.

- I have looped my buttons in my main so more buttons can be easily added with little code changes.

- I know this as I have been adding things this whole time, in each of the sprints. We addressed this in the way the programme has been designed.

- Program elements (colours, buttons and shapes) have all been programed as objects. This means each object is a self contained module piece of code that takes a defined set of parameters to work. Most take basic mouse data and colours to work. All objects are built to a similar structure and compulsorily contain an update function. This means they can appear on the interface using the update call. Management of most of the objects happens in a central place (Control Object). The idea is if you give the program to someone else (an experienced programmer), they could interpret what it does and built a new component. They could remove features and add new features.

- Comments in the code are precise and informative.

Functionality

Does the program work well? My program is free of bugs and loads quickly. It is also able to works on different browsers so I can conclude my program does work well.

How have we addressed this? My program is programmed in pure javascript (almost) so it has a very low file size which allows it to run fast. It has only one graphic. So it loads very quickly. I know this as I have been running it many times. I have tested by using a live web server to store my versions, and have observed loading performance.

To ensure my program works well I have systematically tested it throughout my sprints. I have also tested user interactions.

I have tested position of clicking and direction of dragging. This has included careful checks around boundaries.

The use of the Math.abs function was discovered after testing failures drawing circular shapes (because these objects were being given negative radius values).

Finally, any new components were tested prior to being added to the main program. At various stages in sprints, console logging was used to analyse boundary test functions and to locate problems when bugs appeared.

Trialling and Making Decisions

Can you find examples of exploring different options and making a decision about which is the best solution?

I trialled different options for allowing the user to draw circles on screen giving circular or rectangular guides. For my slider I tried different lengths and size bar and circle to allow the user to best slide the slider and to give the best length for the line width.

I effectively used information from testing and trialling to improve the functionality of the outcome. I trialled interface design colour themes and draw colours.

I found situations where objects drew outside of the drawing area (in spite of having in boudnary parameters). This led to the use of the clip() function.

How did your working process (planning, testing, trialling) lead to the development of a high quality digital outcome? How was the Agile process helpful?

I allowed planning and development to happen together and created repeated cycles of plan develop test. After two or three sprints, I got a much clearer idea of what I could achieve and my aims became much more focused and realistic. The Agile process helped to see the project in a highly simplified form at the start.
